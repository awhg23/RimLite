## 1. 综述 (Overview)

### 1.1 项目背景与核心问题

《轻量版环世界 V0》是一款面向 PC 浏览器的 2D 俯视半实时殖民地模拟 Demo。产品希望保留 RimWorld 式的核心体验：玩家不是单纯控制角色完成任务，而是在开放生存中通过工作调度、需求管理、随机事件和角色后果生成 emergent story。

V0 的核心差异点是加入一个接入云端大模型的 Q 版福瑞小助手。小助手不是普通提示面板，也不是面向玩家的闲聊机器人；它需要理解当前游戏状况，基于摘要上下文自主规划行动，并根据资源、需求、危险和事件状态与游戏内小人对话。它可以自动执行低风险行为，在高风险行为前必须请求玩家确认。

V0 要证明的核心问题：

1. 单个玩家小人 + 一个 AI 福瑞小助手也能形成可持续的开放生存循环。
2. 工作、需求、事件和 AI 规划可以构成轻量版 RimWorld 的主要可玩性。
3. 大模型助手能让殖民地模拟从“被动数值系统”变成“可对话、可协作、会主动判断局势”的陪伴式生存体验。

### 1.2 目标用户与核心场景

目标用户：

1. 喜欢 RimWorld、Dwarf Fortress、Oxygen Not Included 等殖民地模拟或生存经营游戏的玩家。
2. 喜欢轻策略、角色陪伴、AI 角色互动和 emergent story 的 PC Web 玩家。
3. 黑客松、原型评审或早期试玩场景中的体验者。

核心场景：

玩家在浏览器中创建一个小人，进入随机生成的小地图。地图上有基础资源、可采集物、危险点和初始营地。玩家通过暂停下指令安排采集、搬运、建造、进食、休息等工作。福瑞小助手会读取当前局势，和小人进行上下文相关对话，自动执行低风险协助，并在危险行为前请求玩家确认。AI 讲述者持续触发压力、威胁、机会事件，事件结束后留下资源变化、心情变化、伤痕、对话记录或失败后果，游戏继续开放生存。

完整 Demo 的目标试玩时长为 30 分钟。玩家从零扎营，在 40x40 随机格子地图里经历 6-8 个事件，并在硬核失败规则下尽量维持小人、福瑞助手和营地的生存。验收以自由试玩反馈为主，局后问卷记录体验评价，同时保留关键路径检查项。

### 1.3 核心业务流程 / 用户旅程地图

1. **创建小人** - 玩家设置小人的名字、外观、少量属性、特质和技能倾向。
2. **进入随机营地** - 系统生成随机地图、初始营地、基础资源、可采集物、危险点和事件生成点。
3. **观察状态** - 玩家查看小人、福瑞助手、需求、资源、工作队列和当前事件状态。
4. **暂停下指令** - 玩家暂停游戏，安排采集、搬运、建造、休息、进食或事件应对。
5. **助手自主规划** - 福瑞助手基于摘要上下文生成短期计划、和小人对话，并执行低风险行为。
6. **讲述者触发事件** - 系统触发压力、威胁、机会三类事件，改变资源、环境、危险或奖励。
7. **确认高风险行动** - 当助手计划战斗、探索、消耗稀缺资源或进入危险区域时，玩家必须确认或拒绝。
8. **沉淀事件后果** - 事件产生资源、伤病、心情、关系、对话记录、建筑损失或失败状态。
9. **开放生存循环** - 游戏不设固定通关终点，玩家持续经营直到失败或主动结束。

## 2. 产品范围与非目标

### 2.1 V0 必做范围

| 模块 | V0 范围 |
| --- | --- |
| 平台 | PC Web Demo，键鼠操作 |
| 架构 | React + Node 前后端分离，SQLite 保存账号和存档 |
| 视角 | 2D 俯视 |
| 节奏 | 半实时推进，支持暂停、恢复、倍速 |
| 时长 | 30 分钟开放生存 Demo，30 分钟后不强制结束，可触发局后问卷 |
| 角色 | 一个玩家创建的小人，一个 AI 福瑞小助手 |
| 账号 | 邮箱密码注册/登录 |
| 存档 | 多存档槽，支持手动保存、自动保存、读取、删除 |
| 地图 | 40x40 随机格子地图，包含出生安全区、资源区、危险区、可建造区、事件生成点 |
| 工作 | 采集、搬运、建造、进食、休息、基础事件应对 |
| 需求 | 饥饿、睡眠、心情 |
| 资源 | 食物、木材、石材、药品四资源闭环 |
| 建造 | 墙、门、床/睡点、储物区、篝火/简易工作点，包含轻量房间系统 |
| 房间 | 封闭性、床、储物、温度/舒适度简化评分，影响睡眠和心情 |
| 战斗 | 轻量身体部位：头、躯干、左/右臂、左/右腿，支持受伤、昏迷、治疗、死亡 |
| 事件 | 压力、威胁、机会各 3 个事件，共 9 个事件池，30 分钟内触发 6-8 次 |
| AI 助手 | 云端大模型规划、游戏内对话、JSON 动作列表、低风险自动执行、高风险确认、规则 AI 兜底 |
| 反馈 | 失败、主动退出或达到 30 分钟后展示局后问卷 |
| 结局 | 开放生存，硬核失败后展示原因和结算 |

### 2.2 V0 非目标

1. 不做器官、义体、复杂疾病、详细伤口类型和医疗手术。
2. 不做完整 RimWorld 级别的财富威胁公式、派系外交、任务链、完整科技树和复杂生产链。
3. 不做多人游戏、联网协作、Mod 系统、DLC 式扩展。
4. 不做跨局 AI 记忆，福瑞助手只拥有本局短记忆。
5. 不把福瑞助手与玩家自由聊天作为主功能；V0 对话主要发生在助手与游戏小人之间，并围绕游戏局势展开。
6. 不做模型直接控制游戏状态，不做自然语言任意行动解析。
7. 不要求参考图逐像素复刻，只保留浅黄绿色、Q 版、可爱、治愈、陪伴感等视觉方向。

## 3. 核心规则

### 3.1 时间与控制

1. 游戏默认半实时推进。
2. 玩家可以暂停、恢复、调整倍速。
3. 暂停时玩家可以查看状态、安排工作、处理高风险确认。
4. 暂停期间需求和事件不继续推进；恢复后角色继续执行队列。
5. 倍速只改变推进速度，不改变规则结果。

### 3.2 小人与助手

1. 玩家小人由玩家创建，是主要殖民者。
2. 福瑞小助手是固定伙伴，视觉方向参考用户提供图片：浅黄绿色 Q 版福瑞、可爱、治愈、陪伴感。
3. 小人和助手都拥有需求状态；助手至少需要参与饥饿、睡眠、心情相关判断。
4. 助手可根据当前局势对小人说话，话题包括资源短缺、危险临近、任务完成、需求低落、机会事件。
5. 助手可以执行低风险行为，但不应绕过玩家高风险确认。

### 3.3 工作系统

1. 工作是角色行动的基本单位。
2. V0 工作类型包括采集、搬运、建造、进食、休息、避险、基础事件应对。
3. 玩家可为小人下达工作指令。
4. 助手可自动选择低风险协助工作。
5. 工作失败时必须给出用户可理解原因，例如路径不可达、资源不足、角色太饿、危险区域未确认。

### 3.4 需求系统

1. 饥饿会随时间下降，过低时角色优先进食或出现风险提示。
2. 睡眠会随时间下降，过低时角色工作效率降低或主动寻求休息。
3. 心情受饥饿、睡眠、事件结果、受伤、助手对话和环境影响。
4. 需求危险是助手规划触发时机之一。

### 3.5 事件系统

1. AI 讲述者定期评估是否触发事件。
2. V0 事件分为三类：
   - 压力事件：缺粮、天气恶化、建筑损坏。
   - 威胁事件：野兽靠近、敌意访客、小型袭击。
   - 机会事件：空投物资、商人路过、求助信号。
3. 事件必须有开始提示、当前状态、应对方式和结束后果。
4. 事件后果进入开放生存循环，影响资源、需求、心情、伤痕、建筑或故事记录。

### 3.6 AI 助手系统

1. 模型接入默认使用云端大模型 API，PRD 不绑定具体供应商。
2. 触发时机：
   - 事件发生。
   - 任一角色需求进入危险区间。
   - 助手或小人完成关键任务。
   - 每隔一段游戏时间进行轻量复盘。
3. 输入上下文为摘要上下文，包括：
   - 地图摘要。
   - 角色状态。
   - 资源库存。
   - 当前事件。
   - 最近对话。
   - 可用行动列表。
   - 风险标记。
4. 输出内容包括：
   - 短期计划。
   - 对小人的对话内容。
   - 建议行动。
   - 是否需要玩家确认。
5. 模型必须输出结构化 JSON 动作列表，不能只输出自然语言计划。
6. 游戏只执行动作白名单中的行动，例如 gather、haul、build、eat、sleep、avoid、assist_build、warn、request_confirm。
7. 可自动执行低风险行动：采集、搬运、跟随、吃饭、休息、避险、低风险建造协助。
8. 需玩家确认的高风险行动：战斗、冒险探索、消耗稀缺资源、拆除关键建筑、让小人进入危险区域。
9. 模型不可用、超时或返回非法计划时，助手切换到规则 AI 兜底，继续执行吃饭、休息、跟随、避险等基础行为。
10. 助手拥有本局短记忆，记录最近事件、对话、玩家选择和助手计划；读取其他存档或跨局偏好不属于 V0。

### 3.7 30 分钟体验曲线

| 时间段 | 体验目标 | 主要系统压力 |
| --- | --- | --- |
| 0-5 分钟 | 从零扎营，理解操作 | 采集食物和木材，建睡点，熟悉暂停下指令 |
| 5-15 分钟 | 建立基础房间和储物 | 墙、门、床/睡点、储物区、篝火/简易工作点 |
| 15-25 分钟 | 应对威胁和资源压力 | 威胁事件、身体部位受伤、药品消耗、房间舒适度 |
| 25-30 分钟 | 复合事件或失败结算 | 资源紧张、心情下降、助手规划、高风险确认 |

1. Demo 达到 30 分钟后不强制结束，玩家可继续开放生存。
2. 到达 30 分钟、失败或主动退出时，系统展示局后问卷。
3. 30 分钟内事件目标为触发 6-8 次，其中至少包含压力、威胁、机会三类各 1 次。

### 3.8 地图与资源规则

1. 地图为 40x40 随机格子地图。
2. 地图必须包含出生安全区、资源区、危险区、可建造区、事件生成点。
3. 出生安全区必须可通行，并且至少能到达初始食物、木材和可建造区域。
4. 开局物资紧张但可活：玩家必须尽快采集食物、建睡点和储物区。
5. V0 资源只包含食物、木材、石材、药品。
6. 食物用于进食和部分事件消耗；木材、石材用于建造；药品用于治疗身体部位伤害。

### 3.9 建造与房间规则

1. V0 建筑包括墙、门、床/睡点、储物区、篝火/简易工作点。
2. 房间由墙和门形成封闭区域。
3. 房间评分由封闭性、是否有床/睡点、是否有储物区、温度/舒适度简化值组成。
4. 有效房间提升睡眠恢复和心情；露天睡眠降低心情并增加事件压力。
5. 建造失败时必须说明原因，例如资源不足、目标格不可建造、路径不可达。

### 3.10 战斗、身体部位与治疗规则

1. 身体部位包括头、躯干、左臂、右臂、左腿、右腿。
2. 战斗命中后对一个身体部位造成轻伤、重伤或致命伤。
3. 手臂受伤影响采集、搬运、建造效率；腿部受伤影响移动；头部和躯干重伤会显著提高昏迷或死亡风险。
4. 药品可用于治疗受伤部位，治疗能降低死亡风险并逐步恢复状态。
5. 小人死亡或小人与助手均无法完成基础维持时，进入硬核失败结算。

### 3.11 事件池规则

| 类别 | 事件 | 目标 |
| --- | --- | --- |
| 压力 | 食物腐坏 | 迫使玩家补充食物或改变储物策略 |
| 压力 | 寒夜/热浪 | 检验房间、篝火和休息安排 |
| 压力 | 建筑损坏 | 消耗木材/石材并打断舒适循环 |
| 威胁 | 野兽靠近 | 检验避险、搬运食物和高风险确认 |
| 威胁 | 敌意访客 | 检验战斗、撤退和身体部位受伤 |
| 威胁 | 小型袭击 | 检验房间防护、治疗和助手规划 |
| 机会 | 空投物资 | 提供资源补给，但可能出现在危险区域 |
| 机会 | 商人路过 | 用资源交换食物、药品或建材 |
| 机会 | 求助信号 | 提供奖励选择，但可能引入高风险行动确认 |

1. 讲述者 30 分钟内触发 6-8 个事件。
2. 同类高压事件不得连续触发超过 2 次。
3. 威胁事件触发前必须确保玩家有至少一种可理解的应对方式。

### 3.12 账号与多存档规则

1. 玩家使用邮箱密码注册和登录。
2. 登录后进入存档列表。
3. 玩家可以创建新存档、读取存档、手动保存、删除存档。
4. 当前局固定间隔自动保存到对应存档的 autosave 快照。
5. 删除存档前必须二次确认。
6. 刷新页面后应能恢复登录态，并从最近存档或存档列表继续。

### 3.13 局后问卷规则

1. 达到 30 分钟、失败结算或玩家主动退出时展示局后问卷。
2. 问卷字段包括整体评分、是否想继续玩、最喜欢的点、最困惑的点、自由反馈。
3. 玩家可以跳过问卷，但跳过行为也应记录为一次试玩结束。
4. 问卷提交后展示感谢页，并允许返回存档列表。

#### **US-05: 作为玩家，我希望福瑞小助手读取局势并生成行动计划，以便获得像伙伴一样的协助**

* **价值陈述 (Value Statement)**:
  * **作为** 玩家
  * **我希望** 小助手能根据地图、资源、需求和事件自主规划
  * **以便于** 它不只是装饰角色，而是真正参与生存

* **业务规则与逻辑 (Business Logic)**:
  1. **前置条件**: 玩家已进入地图，助手处于可行动状态。
  2. **触发条件**:
     1. 事件发生。
     2. 小人或助手需求进入危险区间。
     3. 关键任务完成。
     4. 定时复盘触发。
  3. **操作流程 (Happy Path)**:
     1. 系统汇总摘要上下文。
     2. 助手请求云端大模型生成短期计划。
     3. 系统校验计划是否属于可用行动。
     4. 低风险计划自动执行。
     5. 高风险计划进入玩家确认。
  4. **异常处理 (Error Handling)**:
     1. 模型超时、不可用或返回非法行动时，助手切换规则 AI 兜底。
     2. 兜底期间助手只执行吃饭、休息、跟随、避险等基础行为。
     3. UI 显示“助手暂时只做基础协助”。

```text

+------------------------------------------------------------+
```

* **验收标准 (Acceptance Criteria)**:
  * **场景1: 低风险计划自动执行**

    * **WHEN** 助手触发规划并生成采集/搬运计划

  * **场景2: 模型失败兜底**

### 阶段四：事件与风险确认

#### **US-07: 作为玩家，我希望 AI 讲述者触发压力、威胁、机会事件，以便开放生存持续产生变化**

* **价值陈述 (Value Statement)**:
  * **作为** 玩家
  * **我希望** 游戏持续触发不同类型事件
  * **以便于** 生存过程不是重复采集和建造，而是不断产生需要应对的局势

* **业务规则与逻辑 (Business Logic)**:
  1. **前置条件**: 玩家已进入开放生存循环。
  2. **操作流程 (Happy Path)**:
     1. 讲述者按节奏评估事件触发。
     2. 系统选择压力、威胁或机会事件。
     3. 系统展示事件标题、描述、风险和建议应对方式。
     4. 事件改变地图、资源、危险或奖励。
     5. 事件结束后写入日志并影响后续生存状态。
  3. **异常处理 (Error Handling)**:
     1. 当前地图不满足事件条件时，讲述者改选其他事件。
     2. 连续威胁过密时，系统应避免立即再次触发高压威胁。
     3. 事件目标不可达时，系统应重选目标或转为可达位置。

```text
| 助手：“我不建议阿青一个人过去，我可以先把食物收起来。”    |
+------------------------------------------------------------+
```

  * **场景1: 触发三类事件**
    * **GIVEN** 玩家持续生存一段时间
    * **WHEN** 讲述者触发事件
    * **THEN** 压力、威胁、机会三类事件都存在可触发代表事件
  * **场景2: 事件不可用改选**
    * **GIVEN** 当前地图不满足某事件条件
    * **WHEN** 讲述者尝试触发该事件
    * **THEN** 系统改选其他可用事件或延后触发

#### **US-08: 作为玩家，我希望确认或干预助手的高风险行动，以便保持关键决策的控制权**

* **价值陈述 (Value Statement)**:
  * **作为** 玩家
  * **我希望** 助手执行危险计划前需要我确认
  * **以便于** AI 自主性不会破坏玩家对生存后果的掌控

* **业务规则与逻辑 (Business Logic)**:
  1. **前置条件**: 助手生成了高风险计划。
  2. **高风险行动包括**: 战斗、冒险探索、消耗稀缺资源、拆除关键建筑、让小人进入危险区域。
  3. **操作流程 (Happy Path)**:
     1. 系统暂停或突出显示确认请求。
     2. 确认面板展示行动、原因、风险、预计收益和替代方案。
     3. 玩家选择允许或拒绝。
     4. 允许时助手执行计划；拒绝时助手执行安全替代行动。
  4. **异常处理 (Error Handling)**:
     1. 玩家长时间不处理时，助手不执行高风险行动。
     2. 事件紧急时，助手最多执行避险类安全行动。
     3. 高风险行动目标失效时，确认请求自动关闭并写入日志。

* **页面布局线框图 (ASCII Wireframe)**:

```text
+------------------------------------------------------------+
| 高风险行动确认                                             |
+------------------------------------------------------------+
| 福瑞助手计划：让阿青靠近北侧危险点查看野兽位置             |
| 原因：确认威胁来源，决定是否搬迁营地                       |
| 风险：可能受伤，可能触发战斗                               |
| 收益：提前发现威胁，减少被突袭概率                         |
+------------------------------------------------------------+
| 安全替代：助手先搬走露天食物，并让阿青留在营地             |
+------------------------------------------------------------+
| [允许执行]                         [拒绝，执行安全替代]     |
+------------------------------------------------------------+
```

* **验收标准 (Acceptance Criteria)**:
  * **场景1: 高风险行动请求确认**
    * **GIVEN** 助手计划进入危险区域
    * **WHEN** 系统识别该行动为高风险
    * **THEN** 游戏展示确认面板，且未确认前不执行该行动
  * **场景2: 玩家拒绝**
    * **GIVEN** 确认面板正在展示
    * **WHEN** 玩家点击拒绝
    * **THEN** 助手放弃高风险行动并执行安全替代行动

### 阶段五：后果与开放生存

---

#### **US-09: 作为玩家，我希望事件后果进入开放生存循环，以便每次选择都留下故事痕迹**

* **价值陈述 (Value Statement)**:
  * **作为** 玩家
  * **我希望** 事件结束后影响资源、心情、伤痕、建筑或日志
  * **以便于** 开放生存不是重置式关卡，而是连续积累的故事

* **业务规则与逻辑 (Business Logic)**:
  1. **前置条件**: 任一事件进入结算。
  2. **操作流程 (Happy Path)**:
     1. 系统根据玩家和助手应对结果结算事件。
     2. 事件后果改变资源、需求、心情、角色状态、建筑或危险状态。
     3. 系统记录事件日志和关键对话。
     4. 若殖民地仍可继续，游戏返回开放生存循环。
     5. 若失败条件达成，展示失败原因和结算。
  3. **异常处理 (Error Handling)**:
     1. 若事件结算数据缺失，系统使用最小后果并写入日志。
     2. 若角色均无法行动，系统进入失败结算。
     3. 若资源为负数，系统修正为 0 并记录资源耗尽。

* **页面布局线框图 (ASCII Wireframe)**:

```text
+------------------------------------------------------------+
| 事件结算：北侧灌木里的动静                                 |
+------------------------------------------------------------+
| 结果：成功避开野兽，营地没有受损                           |
| 资源变化：食物 -1，木材 0                                  |
| 状态变化：阿青心情 -5，助手心情 +3                         |
| 故事记录：助手说服阿青没有贸然靠近危险点                   |
+------------------------------------------------------------+
| [继续生存]                         [查看日志]              |
+------------------------------------------------------------+
```

* **验收标准 (Acceptance Criteria)**:
  * **场景1: 成功回到开放生存**
    * **GIVEN** 事件已被成功应对
    * **WHEN** 玩家点击继续生存
    * **THEN** 游戏返回主界面，事件后果保留在当前局中
  * **场景2: 失败结算**
    * **GIVEN** 小人无法继续行动且助手无法完成基础维持
    * **WHEN** 系统结算事件或需求崩溃
    * **THEN** 游戏展示失败原因和本局记录

#### **US-10: 作为玩家，我希望使用邮箱密码登录，以便管理自己的多个生存存档**

  * **作为** 玩家
  * **我希望** 注册并登录账号
  * **以便于** 在不同设备刷新或重进后继续访问自己的存档

* **业务规则与逻辑 (Business Logic)**:
  1. **前置条件**: 玩家打开 Demo。
  2. **操作流程 (Happy Path)**:
     1. 系统展示登录/注册入口。
     2. 玩家输入邮箱和密码。
     3. 注册成功后自动进入存档列表，或登录成功后进入存档列表。
     4. 系统保持登录态，刷新后仍能识别当前用户。
  3. **异常处理 (Error Handling)**:

     2. 密码不符合要求时提示“密码至少 8 位”。
     3. 登录失败时提示“邮箱或密码错误”。

#### **US-11: 作为玩家，我希望管理多个存档槽，以便保存和继续不同生存局**

* **价值陈述 (Value Statement)**:
  * **作为** 玩家
  * **我希望** 创建、读取、手动保存、自动保存和删除多个存档
  * **以便于** 保留不同开局、不同事件结果和不同试玩进度

* **业务规则与逻辑 (Business Logic)**:
  1. **前置条件**: 玩家已登录。
  2. **操作流程 (Happy Path)**:
     1. 系统展示存档列表。
     2. 玩家可创建新存档或读取已有存档。
     3. 游戏中玩家可手动保存到当前存档槽。
     4. 系统按固定间隔自动保存当前局 autosave 快照。
     5. 玩家可删除存档，删除前必须二次确认。
  3. **异常处理 (Error Handling)**:
     1. 存档读取失败时提示“存档读取失败，请重试”。
     2. 手动保存失败时提示“保存失败，当前进度仍在本地运行中”。
     3. 删除存档确认前不得真实删除。

* **页面布局线框图 (ASCII Wireframe)**:

```text
| 北林实验局    06:15      手动保存 2小时前  [继续] [删除]   |

+------------------------------------------------------------+

+------------------------------------------------------------+
```

* **验收标准 (Acceptance Criteria)**:
  * **场景1: 创建新存档**
    * **GIVEN** 玩家已登录并位于存档列表
    * **WHEN** 点击创建新存档
    * **THEN** 系统创建新存档并进入创建小人流程
  * **场景2: 自动保存**
    * **GIVEN** 玩家正在进行开放生存
    * **WHEN** 到达自动保存间隔
    * **THEN** 系统保存当前游戏状态到该存档的 autosave 快照
  * **场景3: 删除确认**
    * **GIVEN** 玩家点击删除存档
    * **WHEN** 未完成二次确认
    * **THEN** 系统不得删除该存档

#### **US-12: 作为开发团队，我希望在试玩结束后收集反馈，以便判断 Demo 是否值得继续迭代**

* **价值陈述 (Value Statement)**:
  * **作为** 开发团队
  * **我希望** 在失败、主动退出或达到 30 分钟后收集局后问卷
  * **以便于** 了解玩家是否理解核心循环、喜欢 AI 助手，以及哪些地方最困惑

* **业务规则与逻辑 (Business Logic)**:
  1. **前置条件**: 玩家已进行至少一局游戏。
  2. **触发条件**:
     1. 玩家达到 30 分钟试玩时长。
     2. 玩家硬核失败。
     3. 玩家主动退出当前局。
  3. **操作流程 (Happy Path)**:
     1. 系统展示局后问卷。
     2. 玩家填写整体评分、是否想继续玩、最喜欢的点、最困惑的点、自由反馈。
     3. 玩家提交后系统展示感谢页。
     4. 玩家可返回存档列表。
  4. **异常处理 (Error Handling)**:
     1. 玩家可跳过问卷。
     2. 问卷提交失败时提示“反馈提交失败，可稍后重试或跳过”。
     3. 跳过问卷不阻止返回存档列表。

```text
| [______________________________________________________]   |

| [______________________________________________________]   |
+------------------------------------------------------------+
| [提交反馈]                                      [跳过]     |
+------------------------------------------------------------+
```

* **验收标准 (Acceptance Criteria)**:
  * **场景1: 达到 30 分钟展示问卷**
    * **GIVEN** 玩家已生存 30 分钟
    * **WHEN** 系统触发试玩反馈
    * **THEN** 展示局后问卷，并允许提交或跳过
  * **场景2: 失败后展示问卷**
    * **GIVEN** 当前局进入硬核失败结算
    * **WHEN** 玩家查看结算
    * **THEN** 系统提供局后问卷入口
